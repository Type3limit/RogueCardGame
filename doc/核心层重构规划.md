# 核心层重构规划

> 生成时间: 2026-04-17
> 定位: **战略路线图**（核心层重构的唯一权威文档）
> 读者: 核心层改动实施者、需要新增卡牌/植入体机制的内容作者

---

## 0. TL;DR

当前核心层**不是**传统的"巨型状态机"，而是**Factory + Action Queue + Power Hooks 三叉架构**：

- `CardEffectFactory._registry` —— 50+ 条 `type → ICardEffect` 映射（字典式 switch）
- `ActionManager` —— STS 风格动作队列
- `AbstractPower` —— 固定钩子墙（`AtTurnStart` / `OnDealDamage` / `OnCardPlayed` …）

三者**各自有扩展点，但彼此不通**。结果是：每新增一个效果/机制，都要在 **CardEffectData → CardEffectFactory → ICardEffect 子类 → GameAction → AbstractPower 钩子 → CombatManager 消费点 → ImplantSystem.GetTotalBonus → Power.GetDescription** 这条链上各改一小块，形成"扩张式代码腐烂"。

**根治方案**：把 **卡牌效果 / Power 被动 / Implant 加成 / Status / 事件** 统一为同一个声明式原语：

```
Behavior = { Trigger, Condition?, Effect[], TargetSelector? }
```

核心层只保留两件事：**触发分发**（CombatEventDispatcher）与 **效果执行**（单一 ActionManager 路径）。新卡、新植入体、新机制 **99% 在 JSON 里完成**；C# 代码只在需要"全新的触发时机"或"全新的原子效果"时扩展。

## 1. 现状诊断

### 1.1 架构实况

```
Card JSON ──► CardEffectData[] ──► CardEffectFactory.Create()
                                            │
                                            ▼
                                  ICardEffect.Execute(ctx)
                                            │
                             ┌──────────────┴──────────────┐
                             ▼                              ▼
                  ActionManager.AddTo*()            fallback callback
                  (正式战斗路径)                     (测试/部分漏网)
                             │
                             ▼
                  GameAction.Execute()
                             │
                             ▼
              foreach(power in powers) power.OnXxx()  ◄── 固定钩子
                             │
                             ▼
              power 内部硬编码：damage += GetStacks(Strength) 等
```

`ImplantSystem` 游离于此链条之外，靠 `CombatManager.GetTotalBonus("xxxName")` **pull 式**散布在各处被硬编码取用。

### 1.2 已识别的结构性症结

| # | 症结 | 具体表现 | 影响 |
|---|------|----------|------|
| S1 | **钩子墙固定** | `AbstractPower` 缺 `OnOverchargeConsumed` / `OnResonanceConsumed` / `ModifyCardCost` / 实际可用的 `OnKill` | WarMachine / ResonanceAmplify / MindSovereign / HealOnKill / FrontlineCommander / ParasiticBond 描述与实现漂移 |
| S2 | **Effect 双执行路径** | 部分 Effect 只走 `DrawCardCallback` / `AddToDiscardCallback` 不入 ActionQueue | ConditionalDraw / Scry / RetrieveFromDiscard / SearchDeck 行为与描述不符 |
| S3 | **Implant pull 消费点发散** | `GetTotalBonus("energy"/"drawPerTurn"/"maxHp"/"hackSpeedBonus"/...)` 在 `CombatManager` 与若干 Effect 里手动取用 10+ 处 | 新植入体加个名字就得在 C# 加一行；植入体无法表达"受击回血"这类事件驱动效果 |
| S4 | **Condition 字符串特例化** | `hasOvercharge` / `lostHpThisTurn` / `hpBelow50%` 各自一条 `if`，仅 `power>=N` 是通用的 | 新条件要动 `ConditionEvaluator` 源码 |
| S5 | **文本-实现无强绑定** | `Power.GetDescription()` 全部手写中文字符串；卡牌 `description` 字段也是人写 | 实现改了忘改文字，或文字改了漏改实现 |
| S6 | **硬编码数值逃逸出 Modify\* 钩子** | `DealDamageAction.Execute` 内联 `damage += Powers.GetStacks(Strength)`，不走 `ModifyAttackDamage` 管线 | 新增"受伤时额外+X"这类 buff 必须改 Action 源码而不是注册 Power |
| S7 | **Target 粒度不足** | `CardEffectContext` 只有一个 `Target/AllTargets`，`SelfTargetWrapper` 是 one-off 特化 | 无法表达"对敌造伤+对己抽牌"这种常见效果，只能新造一个合并 Effect 类 |
| S8 | **Core 含 Godot 依赖** | `GD.PrintErr` 散布在 `CardEffect.cs` / `ConditionEvaluator.cs` | Core 单测受限；破坏分层 |
| S9 | **EventBus 形同虚设** | 只给 UI 用，核心机制不订阅 | 无法承担跨系统触发分发职责 |

## 2. 目标架构

### 2.1 统一原语 Behavior

```csharp
public sealed class Behavior
{
    public required TriggerKind Trigger { get; init; }     // 何时触发
    public Expr? Condition { get; init; }                  // 触发后是否满足条件
    public TargetSelector? TargetSelector { get; init; }   // 目标（可省略，默认继承 ctx）
    public required IReadOnlyList<ICardEffect> Effects { get; init; }
    public int? Priority { get; init; }                    // 多 Behavior 顺序
}
```

**Behavior 是 Card、Power、Implant、Status 共同的运行时单元**：

| 载体 | 产生方式 |
|------|---------|
| Card | Card.OnPlay 时，把 `CardData.Effects[]` 包成 Trigger=OnPlayCard 的 Behavior 执行 |
| Power | `AbstractPower` 持有一组 Behavior，由 Dispatcher 根据 Trigger 召回 |
| Implant | 进入战斗时自动"施加"一个 persistent Power，其 Behavior 即植入体的全部效果 |
| Status (Buff/Debuff) | 与 Power 完全同构，只是带 TicksDown / 显示归类不同 |

### 2.2 Trigger 总线 CombatEventDispatcher

```csharp
public enum TriggerKind
{
    // 回合/阶段
    OnEnterCombat, OnTurnStart, OnTurnEnd,
    OnPlayerPhaseStart, OnEnemyPhaseStart,

    // 卡牌
    OnPlayCard, OnDrawCard, OnDiscardCard, OnExhaustCard,

    // 伤害/治疗
    OnDealDamage, OnTakeDamage, OnBlocked, OnHeal, OnKill, OnDeath,

    // 消耗
    OnOverchargeConsumed, OnResonanceConsumed, OnProtocolConsumed, OnHackApplied,

    // Power 生命周期
    OnApplyPower, OnRemovePower, OnStacksChanged,

    // 管线 (Modify*) —— 返回修改值
    ModifyCardCost,
    ModifyAttackDamage, ModifyDamageTaken,
    ModifyBlockGain, ModifyHealAmount,
}
```

`CombatEventDispatcher` 维护 `Dictionary<TriggerKind, List<Behavior>>`（按 owner 分组）。各 Action 在关键位置只做一件事：

```csharp
// DealDamageAction 内
var dmg = _dispatcher.Pipeline(TriggerKind.ModifyAttackDamage,
              new DamageCtx { Source=src, Target=tgt, Amount=baseDamage });
// ... 应用 dmg 到 target ...
_dispatcher.Fire(TriggerKind.OnDealDamage, new DamageDealtCtx {...});
```

**Modify\* 类触发**走 pipeline 模式返回新值，取代当前 `foreach power in powers: base = p.ModifyAttackDamage(base)` 与 `damage += Powers.GetStacks(Strength)` 的双重混合。

### 2.3 原子 Effect + 正交 TargetSelector

- **原子化**：每个 `ICardEffect` 只做一件物理动作（造伤、加层、抽牌、回能、加 Power）。组合由 JSON 数组完成。
- **Scaling 通用化**：`damage` 支持 `{ scaleFromPower: "Resonance", multiplier: 3 }`，取代专用 `resonanceDamage` / `protocolDamage` / `damagefromhplost` 等 N 个特化类。
- **TargetSelector 正交**：

```csharp
public abstract class TargetSelector { public abstract IReadOnlyList<Combatant> Resolve(BehaviorCtx ctx); }
// SelfSelector, PrimaryTargetSelector, AllEnemiesSelector, RandomEnemySelector,
// FrontRowSelector, BackRowSelector, LowestHpSelector, MarkedSelector, ...
```

每个 Effect 可独立带 selector。彻底消除"对敌造伤+对己抽牌必须合成一个新 Effect 类"的问题。

### 2.4 Condition Expr 树

```csharp
public abstract record Expr
{
    public abstract bool Eval(BehaviorCtx ctx);
}
public sealed record PowerGte(string PowerId, int N, string Subject="Self") : Expr;
public sealed record HpPercentLt(double Pct, string Subject="Self") : Expr;
public sealed record HasPower(string PowerId, string Subject="Target") : Expr;
public sealed record And(IReadOnlyList<Expr> Terms) : Expr;
public sealed record Or(IReadOnlyList<Expr> Terms) : Expr;
public sealed record Not(Expr Term) : Expr;
```

JSON 写法：

```json
{ "if": { "and": [ { "powerGte": ["Resonance", 5] }, { "hpPercentLt": 0.5 } ] } }
```

保留向后兼容：旧字符串（`"resonance>=5"`）由 `ConditionEvaluator.Parse()` 转为 Expr 节点。

### 2.5 单一执行路径

**删除所有 fallback 回调**（`DrawCardCallback` / `AddToDiscardCallback`）。测试路径改为用 in-memory `ActionManager` + 最小 `CombatManager` stub。benefit：

- 消除"正式战斗走 Queue、测试走 callback"带来的真相漂移
- 描述 `X 回合末抽 Y 张` 与 runtime 真抽时机完全一致

### 2.6 文本-实现单源化

- Power / Card 描述 **半自动生成**：模板 + Behavior 参数替换。
- 关键词一致性测试（CI 强制）：从 Behavior 抽取"damage / block / draw / Resonance / Overcharge / …"关键词集，与本地化文本做交集断言，漂移立即失败。
- 允许每张卡 override 文本（用于润色），但必须通过关键词一致性测试。

### 2.7 Implant 归一化

`ImplantData.Effects` → 装配时构造 `PersistentPower(implant.Id)` 注册到 `PlayerCharacter.Powers`。该 Power 的 Behavior 列表就是植入体的全部功能。移除 `ImplantSystem.GetTotalBonus()` 与 `CombatManager` 里所有对它的调用点。

好处：
- 植入体可以表达事件驱动效果（受击回血、击杀抽牌…），与 Power 一致
- 新植入体加效果 = 写 JSON，不用动代码
- UI 显示"当前激活效果"天然统一

### 2.8 Core 去 Godot 依赖

```csharp
// src/Core/Utils/IEffectLogger.cs
public interface IEffectLogger { void Error(string msg); void Warn(string msg); }
public static class EffectLog { public static IEffectLogger? Logger; public static void Error(string m) => Logger?.Error(m); }

// Godot 侧初始化（CombatScene 或 autoload）
EffectLog.Logger = new GodotEffectLogger();
```

## 3. 分期实施计划

> 每一期独立可验证、独立可发布。下期开工前上期必须 `dotnet test` 全绿。

### P0 — 描述-实现对齐热修（无架构改动）
- HealOnKillPower.OnKill、WarMachinePower、ResonanceAmplifyPower、ParasiticBondPower、FrontlineCommanderPower、MindSovereignPower
- ConditionalDrawEffect / ScryEffect / RetrieveFromDiscardEffect / SearchDeckEffect 补 ActionQueue 路径（或降级并改文本，见 §4 决策 2）

**产出**：6 Power 修复 + 4 Effect 路径统一 + 每个 case 的一致性测试。

### P1 — Trigger 总线落地
- 新增 `CombatEventDispatcher`
- 扩 `AbstractPower` 虚钩子到位（`OnOverchargeConsumed` / `OnResonanceConsumed` / `ModifyCardCost` / `OnKill`），但内部改为"Dispatcher 兜一层"调用 —— 旧代码仍能工作，新代码走 Dispatcher
- `OverchargeConsumeAction` / `ConsumeResonanceAction` / `ConsumeProtocolAction` 改触发 `Fire(TriggerKind.OnXxxConsumed, …)`
- 给 `Power` 加 `Source` 字段（`ParasiticBond` 需要）

**产出**：6 坏 Power 的修复从"硬编码调用"升级为"分发器订阅"；关键 Modify\* 管线单测。

### P2 — 统一执行路径
- 删 `CardEffectContext.DrawCardCallback / AddToDiscardCallback / ImplantBonusProvider`（后者在 P5 前暂保留）
- 测试 harness 提供轻量 `CombatManager` + `ActionManager`
- grep `DrawCardCallback|AddToDiscardCallback` 应为 0 命中

### P3 — Condition DSL
- 引入 `Expr` 记录类族
- `ConditionEvaluator.Parse(string)` 向后兼容；新条件统一用 JSON 结构化写法
- 每条现存字符串条件做 golden-parse 测试，确保等价迁移

### P4 — Per-effect TargetSelector
- `SelfTargetWrapper` 一般化为 `TargetSelector` 类族
- JSON schema 加 `target` 字段（已部分存在），支持所有选择器
- 拆解 `resonanceDamage` / `protocolDamage` 等 scaling Effect → `damage + scaleFromPower` 通用形式（保留旧 type 别名一段时间）

### P5 — Implant → Behavior 归一化
- `ImplantData` 装配时生成 `PersistentPower`
- 下架 `ImplantSystem.GetTotalBonus` 消费点（`CombatManager.Initialize` / `StartPlayerPhase` / `HackEffect` 等）
- 所有 9 个现有植入体回归用例

### P6 — 文本自动/半自动生成
- 从 Behavior 生成默认描述模板
- CI 加关键词一致性测试（§2.6）
- 清理漂移的硬编码描述

### P7 — Core 去 Godot + 清收尾
- `IEffectLogger` + `EffectLog` 注入
- 移除 `src/Core` 下所有 `using Godot` 与 `GD.PrintErr`
- 更新 `doc/architecture.md` 反映新分层

## 8. 写给内容作者：P5 以后加新卡/新植入体长什么样

**目标形态**：无需动 C#，只在 JSON 里声明。

示例：新植入体"镜像核心"——每次打出费用 0 的卡，额外回 1 能量，每回合最多触发 2 次。

```json
{
  "id": "mirror_core",
  "name": "镜像核心",
  "slot": "core",
  "behaviors": [
    {
      "trigger": "OnPlayCard",
      "condition": { "and": [
        { "cardCostEq": 0 },
        { "counterLt": ["mirrorCore.thisTurn", 2] }
      ]},
      "effects": [
        { "type": "gainEnergy", "value": 1, "target": "self" },
        { "type": "incrementCounter", "key": "mirrorCore.thisTurn" }
      ]
    },
    {
      "trigger": "OnTurnStart",
      "effects": [ { "type": "resetCounter", "key": "mirrorCore.thisTurn" } ]
    }
  ]
}
```

只要 `OnPlayCard` / `gainEnergy` / `incrementCounter` 等原子已存在，就**零代码改动**上线。
